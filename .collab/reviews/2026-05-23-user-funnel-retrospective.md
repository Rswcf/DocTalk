# DocTalk 用户复盘:为什么 109 个用户、几乎零付费、DAU 极低

**日期:** 2026-05-23
**数据源:** 生产 Postgres(Railway,只读查询)+ 后端代码定位
**作者:** Claude(systematic-debugging 流程,先定位根因再谈方案)

---

## 0. 一句话结论

**这不是定价问题,而是"留存 + 大文档 RAG 检索质量"问题。** 唯一一个付费用户当天退订,根因是产品在她最看重的场景(大文档问答)上根本没交付价值。**同一个底层缺陷,同时制造了三个现象:付费即退、激活后流失、DAU 上不去。** 三者是同一个病的不同症状。

## 2. 留存悬崖 —— DAU 低的直接原因

| 用户活跃的"不同天数" | 人数 |
|---|---|
| 只活跃 1 天(再没回来) | **39** |
| 2 天 | 6 |
| 3–5 天 | 2 |
| 6+ 天 | **1** |

- 48 个聊过天的用户里,只有 **16 人(33%)** 在注册当天之后回来过;只有 **1 人**真正持续使用(活跃 ≥6 天)。
- **WAU = 6,MAU = 35,DAU 在 1–5 之间波动。**
- 注册 → 首条消息中位数仅 2.7 分钟(说明上手不是问题),问题全在**第一次用完之后不回来**。

> **漏水桶模型:** 每周拉来 3–5 个真实用户,但 ~80% 第一天就流失。无论怎么拉新,DAU 永远是平的低位。没有留存 → 感受不到价值 → 没人付费。"零付费"与"低 DAU"同源。

## 3. 铁证:唯一付费用户的完整 15 分钟(2026-05-06)

**用户:** `755788ef…`,准备**捷克医学国家考试**的学生。
**文档:** 492 页、28.6MB 的捷克语题库 PDF。

**时间线:**

| 时间 | 事件 |
|---|---|
| 14:40 | 注册(Google),送 500 积分 |
| 14:41 | 上传大文件,**撞 free 版 25MB 上限**(`limit_hit: file_size`) |
| 14:41 | 点"升级" → 看价格 → 再点升级 |
| 14:42 | **完成付费 Plus($9.99)** —— 在体验任何价值前就付费,纯为"能上传大文件" |
| 14:43 | 上传 492 页 PDF(成功,status=ready) |
| 15:27 | (已是 Plus)系统仍弹 `activated_free_user` 升级 nudge |
| 15:28–15:43 | 聊天,全程失败(见下) |
| 15:43 后 | **再也没回来** |

**对话实录(捷克语→中文):**
- "找第 80 题并检查错误" → AI:"**我手上的 fragments(片段)里没有第 80 题**"
- "文档开头就有题目列表,去找" → AI:"我研究了片段 1 到 8……"
- "第 80 题在第 350 页" → AI:"**提供的片段不包含第 350 页**"
- "第 350 页有什么" → AI:"我无法确定第 350 页有什么"
- "**什么是 fragment?**"(被内部术语搞懵)
- "**你为什么没有完整的文档?**"
- "你能不基于文档回答吗" → AI:"**不能。**按规则我只能基于提供的文档回答……"
- **"既然上传大文件后片段里只有它的一部分,那能上传大文件的意义是什么?"** ← 客户亲口说出核心缺陷
- **"我今天刚买就发现它毫无用处(k ničemu),能退订吗?"** ← 当天退订请求(且问给了 AI,AI 答不了)
- 15:43 切到 Pro(balanced)模式再试原问题 → 仍失败。

> 这就是"一个人付费、当天说不合适、退订"的全部真相。**退订原因 100% 是 RAG 检索失败,与价格无关。**

---

## 4. 代码层面已定位的根因

### 4.1 检索 top_k 太小,且不随文档大小放大(头号元凶)
- `backend/app/services/corrective_retrieval_service.py:92` — `_plan_limit = max(top_k or 8, 12)`;单文档每步 `top_k=2`,最终最多 **~8–12 个 chunk** 喂给 LLM。
- 492 页文档 → LLM 只看到 8–12 个片段(= 对话里的 "fragments 1 到 8")。用户在 PDF 预览里**亲眼能看到**的内容,AI 却说"不存在" → 对主打"问文档 + 出处引用"的产品,这是最致命的信任崩塌。
- **没有按页/按位置检索**:"第 350 页有什么""找第 80 题"这类导航式查询,纯语义召回必然命中不了(`pages` 表里有页数据,但检索没用上)。

### 4.2 Prompt 泄露内部术语 "fragment",且过度死板
- `backend/app/services/chat_service.py:396` — "Use only the numbered fragments provided by the system. Do not add outside knowledge."
- 整套 prompt 用 "fragment" 措辞 → 捷克语显示成 "fragmenty",用户被迫问"什么是 fragment"。AI 连"能否退款"这种元问题也用"我只能基于文档回答"挡回去,显得笨且机械。

### 4.3 文件大小门槛逼用户"先付费、后体验"
- `backend/app/core/config.py:156-158` — FREE=25 / PLUS=50 / PRO=100 MB。她 28.6MB → **必须买 Plus 才能上传**。在"上传闸口"收费,卖的恰恰是产品交付不了的能力(大文档)。

---

## 5. 次要但值得记的发现

- **~10% 首次上传解析失败**:自有上传 95 个里 10 个 `VECTORIZE_FAILED` + 2 个卡在 `ocr`,集中在 **4/15–4/19**(疑似一次 Qdrant/embedding 事故)。这批用户第一印象即"坏的"。
- **domain mode 试用前付费**:109 个 session 的 `domain_mode` **全为空**,但有 18 次 academic/legal 升级点击 → 没人体验过就被要求付费。
- **付费后仍弹升级**:该付费用户付费后被弹了 4 次 `activated_free_user` nudge。
- **paywall 太激进**:118 次 paywall 中 117 次是主动 nudge,只有 9 人真正撞限制 → 在价值交付前反复推价格。
- **疑似计费 bug**:付 Plus 后 invoice 只发 **+300** 积分(ledger 实测),按 Plus 应为 3000/月 → Stripe 价格→积分映射可能配错,需单独核实。
- **自助退订未被使用**:退订**没写进 `plan_transitions`**(库里仍 plus/sub=true)→ 用户是在聊天框向 AI 要退款,很可能**没找到自助退订入口**,或由后台 Stripe 手动退。
- **唯一"积分包购买"是 owner 自己**(`reason=purchase` 那条 = 首个内部账号),非真实外部收入。

---

## 6. 把宏观与微观连起来

大文档 RAG 失败(§4.1)不是只坑了那一个付费用户 —— 它**很可能在静默地坑掉大量免费用户**:他们没先付费,所以不会留下"退款请求"的痕迹,只是上传大文档 → 问几句 → 发现答非所问 → 第一天就走。这正好解释了 §2 的"39 人只活跃 1 天"。

**即:同一个 RAG 缺陷,既造成付费即退,也造成留存悬崖。** 修它的杠杆率最高。

---

## 7. 建议修复顺序(待决策)

1. **修大文档检索**:top_k 随文档大小放大 + 混合检索(语义 + 关键词 + 按页)+ 召回不足时先返回文档大纲/告知"只看到了部分,要扫描全文吗"。直接命中付费即退 + 留存悬崖。
2. **重排变现时机**:别在上传闸口收费;先免费聊出价值,再在持续使用/真实撞墙时收费;停掉付费后的升级 nudge。
3. **激活与首跑可靠性**:解析失败加重试 + 友好兜底;domain mode 给免费试用名额。
4. **Prompt 去 "fragment" 化**:召回不足时主动说人话,而非冷冰冰"片段里没有"。
5. **核实计费**:Plus 月度积分只发 300 的问题。

# 附录 B:全部 47 个真实用户对话复盘(owner 排除)

## B.0 留存底数
- 发过消息的真实用户:**47**;注册日后回访:15(32%);一次就走(≤2 条且不回头):**21**;上传了却一条没问:13(~10 个是解析失败受害者);**首条消息没收到任何回复:4**(U07/U10/U19/U36)。
- 10 个 `VECTORIZE_FAILED` 解析失败用户**不在**"聊过天"集合里 → 解析失败=直接劝退,沉淀在"上传没问"的 13 人中。

## B.1 用户画像(关键)
全球**学生/研究者**,语言极散(PT/ES/FR/AR/VI/ID/UR/IT/ZH/CS/DE),文档多为**大部头学术材料**(462p 生物书、492p 医学题库、443p 论文、468p 大众年报…)。需求四类:整本书全文总结/cheatsheet、找特定页/题/术语并引用、帮我写论文/导出 Excel·CSV·PDF、读论文 results&discussion。**正好是 8-fragment RAG 最差的场景 → 供需错配。**

## B.2 流失原因(按频次,带原话)
1. **大文档整篇覆盖失败(~10–12 人,头号):** U26(462p,"第18章没有信息")、U18("只有10章吗???")、U47(31条最投入仍"仅凭片段无法完成")、U01/U37/U21/U19/付费用户。
2. **精确出处/页码失败,打脸核心卖点(~6 人):** U01("标注每条引用确切页码"→无)、U21("给我页码"→"片段不含页码")、U03/U17/付费用户。
3. **导出/生成被拒(~5 人):** U28(Excel)、U30(CSV+表格不对)、U44(下载)、U47(可下载PDF)。
4. **过度死板"只能基于文档"人设(~5 人,"帮我写"人群):** U47("在片段之外帮我写"被拒)、U09("我不满意")、U08、付费用户("能不基于文档回答"→"不能")。
5. **扫描件/非拉丁文解析成乱码(~2 可见+沉默 ocr 群):** U13(乌尔都语"编码/加密乱码")、U38(扫描建筑图纸 14MB"乱码")。
6. **首条无回复=可靠性(4 人,8.5%):** U07/U10/U19/U36,多为大文档首问(U19=216p、U10=160p)疑似 SSE 超时/放弃。
7. **上传/交互困惑(~2 人):** U42("那你对我没用",没搞懂怎么加文档)、U31(粘 Google Drive 链接让 AI 做题)。
8. **产品做对了但一次性需求、无回访钩子(~15–18 人):** U02/U06/U12/U16/U22/U24/U25/U29/U32/U33/U34/U35/U40/U43/U45/U46/U41 —— 答得好、引用对,只是一次作业读完就走。**非质量问题,是留存回路缺失。**

## B.3 产品的甜区(别改坏)
**中小文档(<40p)+ 单点聚焦问题**,跨语言答得好且带正确引用:U02/U16/U24/U41(典型好用户,回访2天)/U45/U25。**甜区=中小文档精确问答,不是大部头整篇处理。**

## B.4 贯穿结论
吸引来的是"带大部头、要做整篇任务"的全球学生,8-fragment RAG 只能服务"小文档单点问答"。**一半被功能劝退,一半即使成功也无回访理由 → 同样压低 DAU。47 段对话里 0 人抱怨价格、0 人用 domain mode。** 付费用户的捷克语对话是这批沉默流失者的"有声版"。
