1. **[Must-fix]** `frontend/src/components/marketing/EdLanguageSelector.tsx:125` and `frontend/src/components/marketing/EdLanguageSelector.tsx:228` — the alternate-language `<a>` elements are only created when `open === true`, then mounted through a client-side portal after a user click. The SSG HTML for `/de/use-cases/lawyers` contains the trigger button, not crawlable locale links. That misses the spec's "crawlable locale switcher (real `<a>` links)" requirement (`.collab/plans/2026-05-24-international-seo-locale-urls-spec.md:24`). Hreflang and sitemap still help discovery, but the switcher itself is not crawlable because crawlers do not click a button to materialize hidden portal content. Render the locale anchors in the initial DOM, even if visually hidden/collapsed with CSS.

2. **[Must-fix]** `frontend/src/components/marketing/MarketingShell.tsx:16`, `frontend/src/components/marketing/EditorialHeaderBase.tsx:36`, and `frontend/src/components/landing/EditorialFooter.tsx:10` — the shared marketing chrome still renders labels from `useLocale()`, not from the URL locale or server translator. `LocaleProvider` starts as English (`frontend/src/i18n/LocaleProvider.tsx:50`) and only later detects localStorage/navigator (`frontend/src/i18n/LocaleProvider.tsx:53`), so direct `/de/use-cases/lawyers` requests get German main content but English nav/footer/language-label text in the initial HTML. The spec explicitly called out this risk and required passing strings/locale into the shell (`.collab/plans/2026-05-24-international-seo-locale-urls-spec.md:111`). Pass `locale` and server-resolved chrome strings through `MarketingShell`, or introduce an initial URL-locale client provider for localized marketing routes.

3. **[Should-fix]** `frontend/src/app/use-cases/lawyers/LawyersContent.tsx:101` and `frontend/src/app/use-cases/lawyers/LawyersContent.tsx:105` — localized pages still inject hard-coded English prose into the main article: "Resources like..." and "Platforms like...". On `/de/use-cases/lawyers`, these sentences appear inside otherwise German body copy. Move those source-link sentences into translation keys, using pre/link/post keys if needed, and update all locale JSON files.

4. **[Should-fix]** `frontend/src/i18n/LocaleProvider.tsx:53` and `frontend/src/i18n/LocaleProvider.tsx:66` — the documented `<html lang="en">` SSR tradeoff is acceptable for Phase A, but the client-side mutation still follows stored/browser locale instead of the URL prefix. A direct `/de/...` visit from an English browser stays `lang="en"` after hydration, despite the spec saying locale routes should set `document.documentElement.lang` client-side (`.collab/plans/2026-05-24-international-seo-locale-urls-spec.md:83`). This can be fixed without dynamic SSR by deriving URL locale with `usePathname()` in the client provider or in a small marketing-only client effect.

5. **[Should-fix]** `frontend/src/app/use-cases/lawyers/LawyersJsonLd.tsx:21` and `frontend/src/app/use-cases/lawyers/LawyersJsonLd.tsx:40` — the English page output changes beyond added hreflang tags. The previous page emitted a hard-coded Article description and FAQ answers (`.collab/reviews/2026-05-24-i18n-locale-urls-diff.patch:368` and `.collab/reviews/2026-05-24-i18n-locale-urls-diff.patch:350`); the new component emits translated-key values for Article/FAQ JSON-LD. That may be a reasonable cleanup, but it violates the Phase A "existing English routes diff = no change" gate (`.collab/plans/2026-05-24-international-seo-locale-urls-spec.md:101`). Either preserve the previous English JSON-LD exactly or explicitly accept this SEO-output change.

6. **[Nit]** `frontend/src/i18n/routing.ts:59` and `frontend/src/i18n/routing.ts:70` — path matching is exact and unnormalized. `/de/use-cases/lawyers/` splits to `/use-cases/lawyers/`, which is not in `LOCALIZED_PATHS`, so the language selector falls back to English-only/client-toggle mode. `usePathname()` strips query strings and the current Next config probably redirects trailing slashes, so this is not currently a blocker, but a small `normalizeMarketingPath()` that strips one trailing slash and separates query/hash would make the helper safer.

7. **[Question]** `frontend/src/i18n/routing.ts:32` and `frontend/src/app/[locale]/layout.tsx:20` — `LOCALIZED_PATHS` only includes `/use-cases/lawyers`, and there is no `frontend/src/app/[locale]/page.tsx`, so bare `/de` 404s and the landing page gets no locale URL/hreflang. If the implementation scope was deliberately narrowed to lawyers-only, that is acceptable. If the checked-in spec remains authoritative, Phase A still misses the landing-page gate and definition of done (`.collab/plans/2026-05-24-international-seo-locale-urls-spec.md:91` and `.collab/plans/2026-05-24-international-seo-locale-urls-spec.md:116`).

Overall verdict: **SHIP-AFTER-MUSTFIX**. The hreflang/canonical maps themselves look reciprocal, `x-default` to English is the right default, Next's metadata/sitemap shape supports these language alternates, and I do not see `[locale]` shadowing an existing concrete root route. The blockers are that the crawlable switcher is not actually present in initial HTML and the shared chrome is still client-locale English on locale URLs.
