# Quote Finder + Evidence Board - Codex Adversarial Review R1

Date: 2026-06-12  
Reviewer: Codex  
Plan reviewed: `.collab/plans/2026-06-12-quote-finder-evidence-board.md`

## Executive Verdict

**BLOCK core architecture until D1/D3 are corrected.** The product direction is strong, but the plan is built on two false codebase assumptions:

1. **PDF `pages.content` does not hold raw PyMuPDF page text.** The PDF parse path persists `Page.content=None`; only non-PDF extractors populate it. See `backend/app/workers/parse_worker.py:215-217`, `backend/app/workers/parse_worker.py:237-240`, and `backend/app/workers/parse_worker.py:407-418`.
2. **Persisted chunk bboxes are not span-addressable.** They are a list of `{page,x,y,w,h}` dicts with no sentence text, sentence index, char offsets, or raw-page offsets. See `backend/app/services/parse_service.py:526-530` and `backend/app/services/parse_service.py:918-942`.

As written, the verifier has no authoritative PDF page text to verify against and no persisted mapping from verified quote span to highlight geometry. Quote Finder can still ship, but v1 must first add/backfill a verification substrate or explicitly scope to chunk-text verification with weaker trust language.

## Codebase Facts Checked

- `Page.content` is nullable in the model (`backend/app/models/tables.py:102-119`) and was added as nullable only (`backend/alembic/versions/20260208_0011_add_page_content.py:16-17`).
- Non-PDF extraction stores original extracted page text in `extracted_content_map` (`backend/app/workers/parse_worker.py:215-240`), but PDF extraction never fills that map (`backend/app/workers/parse_worker.py:273-385`). The shared page persistence then writes `content=raw_content...` (`backend/app/workers/parse_worker.py:407-418`), so PDF rows get `NULL`.
- The document text API explicitly treats `Page.content` as available for newly parsed non-PDF docs and falls back to chunk reconstruction when no page has content (`backend/app/api/documents.py:691-700`, `backend/app/api/documents.py:712-770`).
- Chunk bboxes are JSONB (`backend/app/models/tables.py:128-145`) and are generated by assigning each split sentence the source line bbox (`backend/app/services/parse_service.py:469-487`, `backend/app/services/parse_service.py:526-530`). The bbox dict contains only page and normalized rectangle coordinates (`backend/app/services/parse_service.py:936-942`).
- Existing chat citation payload shape is `ref_index`, `chunk_id`, `page`, `page_end`, `bboxes`, `text_snippet`, `offset`, `confidence_score`, `context_text`, optional document/table fields (`backend/app/services/chat_service.py:207-269`). Frontend maps exactly those fields into `Citation` (`frontend/src/lib/api.ts:70-84`, `frontend/src/types/index.ts:12-25`).
- Existing frontend citation navigation only sets `currentPage`, `highlights`, and `highlightSnippet` from the citation (`frontend/src/store/index.ts:154-164`).
- PDF snippet fallback is limited: it only runs when all bboxes are dummy full-page bboxes (`frontend/src/components/PdfViewer/PageWithHighlights.tsx:64-77`). Text viewer snippet matching exists separately (`frontend/src/components/TextViewer/TextViewer.tsx:90-116`, `frontend/src/components/TextViewer/TextViewer.tsx:258-266`).
- Current retrieval is chunk-based semantic Qdrant + lexical `ILIKE` over `Chunk.text` and `Chunk.section_title` (`backend/app/services/retrieval_service.py:236-289`, `backend/app/services/retrieval_service.py:291-340`). The dynamic breadth cap is 24 chunks for single documents (`backend/app/services/corrective_retrieval_service.py:92-114`).
- Quote/source/verbatim intent already routes through the normal RAG answer path, not a distinct verified pipeline (`backend/app/services/action_planner.py:18-50`, `backend/app/services/action_planner.py:218-225`; test at `backend/tests/test_action_planner.py:73-77`).
- `rapidfuzz` and `citeproc-py` are not current backend dependencies; `python-docx` is (`backend/requirements.txt:1-27`).

## D1. Verbatim Guarantee

**Verdict: BLOCK.**

The guarantee is currently impossible for PDFs because the proposed verification target, `pages.content`, is empty for the primary target format. The plan says "pages store RAW PyMuPDF text"; actual code stores no PDF page text at all (`backend/app/workers/parse_worker.py:407-418`). For non-PDF docs, `Page.content` is extractor text, not raw source layout text (`backend/app/services/extractors/docx_extractor.py:35-63`, `backend/app/services/extractors/pptx_extractor.py:30-77`).

Required changes:

- Persist PDF page text during parse. Either store raw `page.get_text("text")` per page or reconstruct from `PageInfo.blocks` with a documented, tested normalization contract. Do not call it raw if it is line-cleaned/chunk-cleaned text.
- Backfill or gate existing PDFs. Existing production PDFs have `Page.content=NULL`; verifier must either reparse/backfill before Quote Finder is enabled or fall back to a clearly labeled weaker chunk-text verifier.
- Store verification offsets: `page_start_offset`, `page_end_offset`, `page_text_hash` or equivalent. Without offsets, saved quotes cannot be reliably revalidated after reparses.
- Treat `Document.text_quality` as nullable/unknown. It is add-only and nullable (`backend/alembic/versions/20260524_0032_add_document_parse_metadata.py:22-26`) and only written at successful embedding finalization (`backend/app/workers/parse_worker.py:579-585`).
- Add quote-specific accounting for discarded/flagged proposals. Existing claim verification is answer-support verification, not quote verification (`backend/app/services/claim_verifier_service.py:156-257`).

## D2. Normalization Pipeline

**Verdict: REVISE.**

The proposed normalization is reasonable, but it is much more complex than the plan budgets. The codebase currently has only a limited inline normalizer for parse (`backend/app/services/parse_service.py:882-891`) and lowercase prefix snippet matching in TextViewer (`frontend/src/components/TextViewer/TextViewer.tsx:90-116`). There is no shared index-preserving normalizer.

Required changes:

- Build this as a standalone, heavily tested service before LLM work. Include offset projection tests for hyphen joins, collapsed whitespace, soft hyphens, Arabic marks, CJK punctuation, and repeated phrases.
- Keep exact/normalized/fuzzy matching separate from display. Displaying raw slices only works once raw page text actually exists.
- Add minimum quote length and boundary checks. Fuzzy partial-ratio matching is unsafe on short phrases and repeated boilerplate unless constrained by length, page, and preferably source chunk.

## D3. bbox / Highlight Reuse

**Verdict: BLOCK.**

"Reuse, don't rebuild" is false for verified quote-span highlights. Existing chunk bboxes are sentence-ish/line-level rectangles with no offsets or text attached (`backend/app/services/parse_service.py:469-487`, `backend/app/services/parse_service.py:526-530`). After persistence, a bbox is only `{page,x,y,w,h}` (`backend/app/services/parse_service.py:936-942`), so the server cannot select "sentence bboxes overlapping the verified span" without reconstructing alignment from scratch.

The frontend also does not provide the claimed generic fallback. PDF text-snippet fallback only activates for all-dummy bboxes (`frontend/src/components/PdfViewer/PageWithHighlights.tsx:64-77`), and normal PDF citations with empty/partial bboxes will not snippet-highlight. TextViewer fallback is for text-based viewing (`frontend/src/components/TextViewer/TextViewer.tsx:258-266`).

Required changes:

- For v1, either emit all chunk bboxes and label highlight precision as approximate, or add a span-to-bbox mapper that reopens the PDF/text layer and maps quote offsets to PDF words/lines.
- If using current bboxes, store `source_chunk_id` and `quote_span_in_chunk` so highlighting can at least choose a subset by sentence order after adding sentence metadata.
- Update PDF frontend fallback so a verified quote with no bboxes can use `textSnippet` on the PDF text layer, not only dummy bbox citations.

## D4. Retrieval

**Verdict: REVISE.**

Reusing hybrid retrieval is fine as a first pass, but "over-retrieve ~2x chat" is not enough for quote search. Current single-document dynamic retrieval caps at 24 chunks (`backend/app/services/corrective_retrieval_service.py:92-114`), and lexical search is `ILIKE` over extracted terms, not a phrase index over page text (`backend/app/services/retrieval_service.py:210-230`, `backend/app/services/retrieval_service.py:291-340`). Quote Finder is recall-bound; a missed chunk means a false "no verified quote" result.

Required changes:

- Add deterministic candidate expansion after semantic retrieval: exact/normalized scan of page text or all chunks for extracted key terms/phrases.
- Add pg_trgm or another indexed phrase path earlier than "deferred unless replay fails" if the feature is marketed as quote finding across long PDFs.
- Record `retrieved_count`, `candidate_pages`, `verified_count`, and `discarded_count` per quote search so recall failures are measurable.

## D5. Citation Formatting

**Verdict: REVISE.**

The library choice is plausible, but the confidence level is too high. Official PyPI metadata for `citeproc-py` v0.9.3 (Apr 30 2026) says it is still major version 0/API not stable, passes roughly 60% of relevant citeproc tests, and lists missing features including disambiguation/year-suffix, collapsing, and punctuation-in-quote: https://pypi.org/project/citeproc-py/

The plan acknowledges year suffixes but underestimates the rest. Academic users will notice broken et-al/subsequent/collapsing behavior. Also, `citeproc-py` is not in current requirements (`backend/requirements.txt:1-27`).

Required changes:

- For first paying users, ship a narrower contract: copy quote + page locator + user-editable source metadata, with APA in-text as best-effort.
- Add golden tests against Zotero before exposing MLA/Chicago labels.
- Keep fallback templates, but do not bury them behind a supposedly standards-complete wrapper.

## D6. Bibliographic Metadata

**Verdict: REVISE.**

The flow is too large for the first paid-user loop, and it depends on D1's false page-content assumption for PDFs. First-pages text is not available from `Page.content` for PDFs today. Also, Crossref/OpenLibrary plus LLM extraction introduces network, latency, privacy, and failure-mode surface that is not needed for the core trust contract.

Required changes:

- Start with manual/editable metadata seeded from filename and PyMuPDF metadata. Add DOI/Crossref only after quote verification works.
- Never block quote-card rendering on biblio lookup.
- Cache provenance and user edits explicitly, and make failures visible but non-fatal.

## D7. Exports

**Verdict: REVISE.**

`python-docx` is available (`backend/requirements.txt:20`) and `_sanitize_xml_text` exists (`backend/app/services/export_service.py:23-31`), but current exports are session-message exports, not quote-bank exports (`backend/app/services/export_service.py:34-90`, `backend/app/api/export.py:62-127`). Extraction exports are MD/CSV and are not Plus-gated (`backend/app/api/extractions.py:313-345`). BibTeX/RIS are entirely new surfaces.

Required changes:

- Ship DOCX quote-bank export first. Defer BibTeX/RIS unless user interviews show that reference-manager import is a purchase blocker.
- Implement quote-bank export as a separate endpoint with its own authorization and Plus gate, not by bending chat session export around saved quotes.
- Add export tests for XML-invalid chars, non-ASCII filenames, RTL/CJK quotes, duplicate quotes, and empty note fields.

## D8. Data Model

**Verdict: REVISE.**

The tables are directionally right, but under-specified for production.

Required changes:

- `saved_quotes` needs `source_chunk_id`, `page_start_offset`, `page_end_offset`, `page_text_hash`, `quote_hash`, `deleted_at` or a duplicate strategy, and indexes on `(user_id, document_id, created_at)` plus `(user_id, document_id, quote_hash)`.
- Store `verification_score` and verifier version, not only `verification_tier`.
- Store `bboxes` as nullable or default `[]`; do not make quote saving depend on highlight availability.
- `document_biblio` as a separate table is fine, but add provenance details: identifiers found, provider response metadata, `user_edited_at`, and schema version.
- Enforce access by document ownership everywhere. Existing `can_access_document` is the relevant pattern (`backend/app/api/documents.py:706-710`, `backend/app/api/chat.py:158-168`).

## Open Questions

1. **Credit pricing: use extraction/job-style predebit for explicit Quote Finder.** Quote search is not a chat message; it should create a quote-search job or result record, debit with `ref_type=document_job`/quote-job id, and record `UsageRecord.message_id=None` like extraction does (`backend/app/services/extraction_service.py:554-567`). If a chat message is routed into the quote pipeline, keep the chat UX but account it as quote search internally.
2. **Free saved quote limit: gate on saved quote count, not board count.** There is no board entity in the proposed model. I would start at 30 saved quotes per user across all documents, with exports Plus-gated. Twenty may cut off a thesis user before they feel the workflow is real.
3. **Normal chat quote intent: yes, but only for strict direct-quote intent.** Organic users ask via chat, and the action planner already routes quote/verbatim/source lookups into `CITATION_LOOKUP` (`backend/app/services/action_planner.py:218-225`). Route "give me a direct quote/original wording/verbatim passage" through verified quote search. Do not hijack broad "what is the source?" or ordinary citation-quality questions until confidence is high.
4. **`document_biblio`: separate table.** Keep it separate from `documents`; biblio metadata is optional, mutable, provenance-heavy, and likely to grow. A column would make document rows carry unrelated mutable blobs.
5. **Tier-3 thresholds: 95 auto / 90 flag is not too strict, but add guards.** Auto-accept fuzzy only for sufficiently long quotes, sane length ratio, same page/source chunk, and clean text quality. For short quotes, repeated phrases, or `text_quality <0.75`, exact/normalized only or flagged. The promise is "verified verbatim," not "found something similar."

## Scope Challenge

Not needed for first paying users:

- Full biblio pipeline with Crossref/OpenLibrary/LLM fallback.
- MLA and Chicago in v1.
- BibTeX/RIS exports.
- Programmatic SEO pages before the core workflow is verified in production.
- Collection quote search.
- Notes-heavy Evidence Board UI beyond save/list/edit-note/delete/copy.

Missing but needed:

- PDF page text persistence/backfill/reparse gate.
- Quote verifier unit tests and real-PDF fixtures before UI work.
- Quote search telemetry: no-results, verified/discarded counts, latency, pages scanned, save/export events.
- Access-control tests for quote CRUD/export.
- Idempotency/deduplication for repeated saves.
- Refund/reconcile behavior for quote-search failures.
- i18n plan for all 11 locales for any new UI copy.
- A production replay harness using the retained academic query corpus before calling the trust contract live.
